# 汎用機・スーパーコンピュータのOS

*作成日: 2026-01-17*
*ステータス: 初稿*

## 概要

これまでの議論ではミニコン〜PC系OS（UNIX、VMS、OS/2等）を中心に扱ってきた。本ドキュメントでは、汎用機（メインフレーム）とスーパーコンピュータのOSについて、その系譜と特徴をまとめる。

---

## 汎用機（メインフレーム）

### IBM System/360系（1964年〜）

IBMメインフレームは、現代でも銀行・保険・航空などの基幹システムで稼働し続けている。

```
OS/360 (1966)
   ↓
MVS (1974) - Multiple Virtual Storage
   ↓
OS/390 (1995)
   ↓
z/OS (2000〜現在)
```

#### 特徴

| 特徴 | 説明 |
|------|------|
| **バッチ処理の王者** | 大量トランザクション処理に最適化 |
| **後方互換性の徹底** | 1960年代のコードが今も動く |
| **信頼性最優先** | 99.999%稼働率（年間5分程度のダウンタイム） |
| **独自アーキテクチャ** | UNIX/PCとは完全に別世界 |

#### OS/360の歴史的意義

OS/360は「最初の大規模ソフトウェアプロジェクト」として知られ、Fred Brooksの『人月の神話』の題材となった。ソフトウェア工学という分野が生まれるきっかけとなったプロジェクトでもある。

### VM/CMS - 仮想化の先駆者

```
CP-67 (1967)
   ↓
VM/370 (1972)
   ↓
z/VM (現在)
```

**注目点**: 1960年代に既に完全仮想化を実現していた。

- 各ユーザーに独立した仮想マシンを提供
- ハードウェアの完全なエミュレーション
- 現代のVMware、KVM等の直接の先祖

これは現代の仮想化技術が「新しい発明」ではなく、「再発見」であることを示している。

### 他の汎用機ベンダー

| ベンダー | OS | 特徴 |
|---------|-----|------|
| IBM | z/OS, z/VM | 現役最古参、業界標準 |
| Unisys | MCP | Burroughs系、スタックマシン |
| Unisys | OS 2200 | Univac系 |
| 富士通 | MSP, XSP | IBM互換 + 独自拡張 |
| NEC | ACOS | 独自路線 |
| 日立 | VOS3 | IBM互換 |

## スーパーコンピュータ

### 黎明期（1970-80年代）

```
Cray-1 (1976)
   ↓ 独自OS: COS (Cray Operating System)
   ↓
UNICOS (1985) - UNIX系に移行
```

この時代は各社が独自OSを開発していた：

| ベンダー | OS |
|---------|-----|
| Cray | COS → UNICOS |
| CDC | SCOPE, NOS |
| 富士通 | 独自OS |
| NEC | SUPER-UX |
| 日立 | 独自OS |

独自OSが必要だった理由：
- ベクトル演算の最適化
- 特殊なメモリアーキテクチャ
- バッチジョブ管理の特殊要件

### 現代（2000年代〜）

**結論: ほぼ全てLinux**

```
TOP500 (2023年時点):
- Linux: 100%（実質的に）
- 独自OS: ほぼ消滅
```

#### なぜLinuxに収束したか

1. **コスト削減**
   - 独自OS開発・維持は莫大なコスト
   - Linuxはオープンソースで無料

2. **ソフトウェア資産**
   - 科学計算ライブラリがLinux前提
   - MPI、OpenMP等の並列化フレームワーク
   - Python、R等のデータサイエンスツール

3. **開発者の要求**
   - 研究者がLinux環境を好む
   - 開発・デバッグの容易さ

4. **カスタマイズ性**
   - オープンソースなので自由に改造可能
   - スケジューラやファイルシステムを専用に最適化

#### 日本の主要スパコン

| システム | 稼働年 | OS |
|---------|--------|-----|
| 富岳 | 2021 | Red Hat Enterprise Linux派生 |
| 京 | 2012 | Linux |
| 地球シミュレータ (初代) | 2002 | SUPER-UX (NEC独自) |
| 地球シミュレータ (現行) | - | Linux |

富岳も京もLinuxベース。独自OSの時代は終わった。

## 汎用機とUNIX/PCの比較

| 観点 | 汎用機 (z/OS等) | UNIX/Linux |
|------|----------------|------------|
| 設計思想 | 信頼性・継続性 | 柔軟性・シンプルさ |
| 互換性 | 60年間維持 | 比較的緩い |
| ターゲット | 大企業の基幹業務 | 汎用 |
| 開発体制 | ベンダー主導（クローズド） | コミュニティ/ベンダー混合 |
| コスト | 極めて高価 | 低〜中程度 |
| カスタマイズ | 限定的 | 自由 |

### 汎用機のプロセスモデル

汎用機は「プロセス」という用語を使わないことが多い。

```
UNIX:
  プロセス = 独立した実行単位

z/OS:
  アドレス空間 (Address Space)
    └── タスク (Task)
          └── サブタスク
  ジョブ = バッチ処理の単位
```

用語と階層構造は異なるが、**本質的には同じ問題を解決している**：
- 複数の計算を同時に実行
- 互いに隔離
- リソースを公平に分配

つまり、汎用機もプロセスモデルの変種を採用していると言える。

## 系譜の全体像（更新版）

```
1960年代
├── 汎用機系 ─── OS/360 ───→ MVS ───→ z/OS（現役）
│                  └── VM/370 → z/VM（仮想化の祖）
│
├── 学術系 ──── CTSS → Multics ──┐
│                                │反面教師
├── ミニコン系 ─ UNIX ←──────────┘
│               ├── BSD → macOS
│               └── Linux → Android, スパコン, クラウド
│
├── DEC系 ───── VMS ───→ Windows NT → 現代Windows
│
└── PC系 ────── CP/M → MS-DOS → Windows 9x（消滅）

スパコン系
├── 独自OS時代 (1970-90s): COS, UNICOS, 各社独自
└── Linux収束時代 (2000s-): ほぼ全てLinux
```

### 現代の勢力図

```
サーバー/クラウド: Linux が支配的
デスクトップ:      Windows, macOS, (Linux)
モバイル:          Android (Linux), iOS (Darwin/BSD)
組み込み:          Linux, RTOS各種
スパコン:          Linux
汎用機:            z/OS（IBM独占）
```

Linuxの影響力が極めて大きいことがわかる。

## LOSへの示唆

### 汎用機から学ぶこと

1. **信頼性へのコミットメント**
   - z/OSの99.999%稼働率は驚異的
   - 「動き続ける」ことの価値
   - 関数型の検証可能性でこれに近づけるか？

2. **後方互換性の価値**
   - 60年間動くコード
   - 互換性維持のコストと価値のトレードオフ
   - LOSは新規設計なので、最初から互換性を考慮する必要はない

3. **仮想化は古い技術**
   - 1960年代に既に存在
   - 「新しい」と思われている技術の多くは再発見

### スパコンから学ぶこと

1. **独自路線は維持できない**
   - 最終的に全てLinuxに収束した
   - エコシステムの力に逆らうのは困難

2. **カーネルより上が重要**
   - スパコンの差別化はランタイム・ライブラリ・ツール
   - カーネルはLinuxで十分
   - LOSも「カーネル自体」より「抽象化レイヤー」で勝負すべきか？

3. **ソフトウェア資産の決定的重要性**
   - 使いたいソフトウェアが動かないOSは使われない
   - 既存エコシステムとの接続方法を考える必要がある

### LOSの位置づけを考える

```
汎用機: 信頼性最優先、互換性維持
スパコン: 性能最優先、Linux上で差別化
PC/サーバー: 汎用性、エコシステム
LOS: ???
```

#### 可能性のある差別化軸

1. **安全性・検証可能性**
   - 汎用機の信頼性 + 関数型の検証可能性
   - 「正しさが証明されたOS」という新しい価値

2. **効果の明示性**
   - 副作用が型で表現される
   - 「このプログラムが何をするか」が明確

3. **教育・研究用途**
   - 「OSの本質とは何か」を学ぶための教材
   - プロセスモデルに縛られない思考の訓練

実用OSとしてLinuxと競争するのは現実的ではない。別の価値軸で勝負する必要がある。
